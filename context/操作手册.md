三、V14 重构路线（路线B：轻量配置驱动）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 核心原则

  不做 HookManager、不做 IRelic 接口、不做 EventBus。
  1200 行代码不需要企业级框架。
  目标是"配置驱动"而非"架构驱动"：
    配置数据描述"是什么"
    核心引擎执行"怎么算"
    不在引擎里写 if-boss-xxx 或 if-relic-xxx

■ 文件拆分（3 个文件起步）

  data.js     — 所有配置：Boss定义、圣物定义、卡牌定义、数值常量
  core.js     — 所有逻辑：战斗引擎、结算、状态管理
  ui.js       — 所有渲染：牌面、槽位、血条、动画

  原因：文件多了AI需要跨文件跳转，3个文件每个~500-700行，一个会话上下文装得下。
  等某个文件突破1000行再拆。

■ 三个最少预留

  预留1：结算管道（圣物介入点），约50行改动
    现在攻击结算是一条直线。未来变成：
      resolveAction(type, totalCount, maxLen, context)
        → calcBase → calcPursuit → applyRelicModifiers → applyStatusEffects
    applyRelicModifiers 用配置表而非 if-else。

  预留2：Boss 行为配置表，约80行改动
    现在 switch-case 敌人逻辑变成 BOSS_CYCLE_TEMPLATE 配置数组。
    新Boss = 新配置对象，不动 executeEnemyTurn 执行函数。

  预留3：存档格式（versioned JSON），约30行改动
    {
      version: 1,
      run: { playerHP, shield, turn, ... },
      relics: [],        // 当前持有圣物
      boss: 'sphynx',    // 当前BossID
      deck: { remaining: 240, consumed: 0 },
      meta: {            // 局外数据（见下表）
        totalRuns, clears, bestCombo,
        unlockedBosses, unlockedRelics,
        skins, collection
      }
    }
    meta 字段预留了 P2/P3 所有扩展（排行榜/图鉴/皮肤/局外养成）。

■ 实现顺序

  第一轮：V14 骨架 + 暹罗Boss + 连击核心/扩容核心 2圣物
          跑通"选圣物→打三关→通关/死亡"完整循环
  第二轮：接入全部10 Boss + 12圣物（填配置数据）
  第三轮：P0关键组合测试


四、产品路线图
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  阶段      触发条件          核心目标            持续时长
  ───────────────────────────────────────────────────
  P0·闭环   当前             一局完整可玩战斗      2-3周
  P1·上瘾   D1留存>35%       玩家愿意反复开        4-6周
  P2·养活   广告eCPM达标     单用户收益>获客成本    6-8周
  P3·长出来  DAU稳定增长     排行榜/图鉴/皮肤       持续

■ P0·闭环（现在）

  做：V14重写，3文件数据驱动，第一关教学+二关核心+三关Boss
      1-3只猫猫，6-8个圣物可运行
      选圣物→打3关→通关/死亡→结算
  不做：排行榜、图鉴、皮肤、局外养成、广告接入
  通过标准：3人真玩，至少2人说"再来一局"

■ P1·上瘾

  做：广告复活、双圣物广告、全部10Boss/12圣物、基础数据埋点
      简单的局外进度统计（通关次数/最高连击）
  不做：排行榜、图鉴UI、皮肤系统
  通过标准：D1留存>35%，广告点击率>3%

■ P2·养活

  做：排行榜（微信好友+全服）、图鉴收集、分享裂变
      3-5套卡牌皮肤、广告位优化
  不做：局外数值养成、大型活动
  通过标准：广告eCPM稳定，分享率>5%

■ P3·长出来

  做：皮肤/剑气特效/槽位皮肤/背景主题
      赛季Boss池/新圣物/限定皮肤
      活动地图/限时Boss/排行榜冲刺
      多语言/多平台

■ 设计原则

  1. 每阶段只加一个系统，做完跑通再做下一个
  2. 数据>直觉，P0没验证"再来一局"不进P1
  3. 代码提前预留、但不提前实现（saveData.meta 占位但不写UI）


五、工具链决策
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 当前工具

  主力：OpenClaw + DeepSeek V4 Pro（本会话所用的组合）
  项目单文件1200行，完全够用。
  不需要切 Claude Code／Cursor／Codex。

■ 何时切换

  代码量突破 5000 行 或 文件数突破 10 个 时，再评估是否切 Claude Code。
  届时切换成本极低——因为 V14 已经是配置驱动结构，
  任何 AI 工具都能无痛接入。

■ DeepSeekTUI

  不适合本项目。它是高自由度本地AI终端，适合hacker workflow/CLI工程。
  本项目核心问题是游戏架构设计，不是终端操作效率。

■ 工具使用纪律

  - 代码改动前先 git status，改后立刻 commit
  - 使用完整飞书 doc_token（非 alias），避免 API 404
  - 飞书 create 文档后需等待索引生效才能 append
  - 文件版本管理：每次大改动先 copy 带版本号的新文件
