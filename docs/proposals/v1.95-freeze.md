## 2. 冻结方案

### 目标结构

```
code/
├── content/           ← Content Layer（纯数据，零逻辑）
│   ├── cards.js       CARD_TYPES / DECK_SIZES / BUFF_TYPES
│   ├── bosses.js      BOSSES（10只猫：属性+特质触发器引用）
│   ├── relics.js      RELICS（11个圣物：属性+效果引用）
│   └── stages.js      STAGES（三关流程+无尽模式）
│
├── rules/             ← Rules Layer（纯函数，无DOM/无状态修改）
│   ├── combat.js      伤害公式/连击计算/Buff结算（纯函数，输入→输出）
│   ├── relics.js      圣物效果函数（每个圣物一个纯函数/effect处理器）
│   └── bosses.js      Boss特质处理器（每个trait一个纯处理器）
│
├── runtime/            ← Runtime Layer（状态管理，无DOM操作）
│   ├── state.js        游戏状态工厂 newGame() / 状态读写
│   ├── turn.js         回合流水线（executeTurn / enemyTurn）
│   └── flow.js         流程控制（三关/无尽/圣物选择）
│
├── ui/                 ← View Layer（纯渲染，不修改状态）
│   ├── render.js       主渲染（从状态读数据→画DOM）
│   ├── board.js        牌堆渲染
│   └── panels.js       弹窗/结算面板/log面板
│
└── index.html          入口（加载顺序：content→rules→runtime→ui）
```

### 依赖方向（单向流）

```
content ──→ rules ──→ runtime ──→ ui
  │           │           │         │
  │           │           │         └─ 只能读 state，从不改
  │           │           └─ 调用 rules 纯函数 + 修改 state
  │           └─ 只依赖 content 常量 + 入参
  └─ 最底层，零依赖
```

### 冻结边界规则（红线）

1. **Content 不包含函数逻辑** — data 文件里不能有 `function`、`execute`、`onXxx` 等执行体
2. **Rules 不访问 G 全局状态** — 所有函数签名明确输入→输出，不隐式读 G
3. **Runtime 不碰 DOM** — 不调用 `document.getElementById`、`render()`、`log()`
4. **UI 不修改 G** — 渲染函数只读 G，写操作通过事件→Runtime 回调

## 3. 执行计划

### Sprint 1：解耦 Content（低风险）
- 从 data.js 中移除圣物的 `onInit`/`getMultiplier` 函数体
- 从 data.js 中移除 Boss traits 的 `execute`/`onResolve`/`onTurnStart`/`onTurnEnd` 函数体
- 从 core.js 中移出 STAGES 到 stages.js
- 从 core.js 中移出意图文本映射表

### Sprint 2：解耦 Rules（中风险）
- 抽取伤害公式纯函数到 rules/combat.js
- 抽取连击计算纯函数
- 各圣物效果改为独立纯函数

### Sprint 3：解耦 Runtime → UI（高风险）
- Runtime 不再直接调用 render()
- 改为"状态变更→事件→UI刷新"模式
- log() 改为状态追加而非直接写 DOM

---

## 4. 风险 & 决策点

| 决策 | 选项 A | 选项 B |
|------|--------|--------|
| 拆分粒度 | 每文件一个概念（10+文件） | 按层合并到大文件（3-5个） |
| 模块格式 | 保持 IIFE/var（无需构建） | 迁移到 ES modules（需 build） |
| 兼容性 | 单文件打包版继续维护 | 放弃单文件版 |
| 首次冻结范围 | 全量重构（一次性） | 只做 Content+Rt 两层 |

**建议**：选项A+保持var+继续维护单文件版+先做Sprint1。原因是：
- 不引入构建工具（保持纯 HTML 打开能用）
- Sprint1 风险最小，可立即验证效果
- Sprint2/3 在此基础上增量推进
